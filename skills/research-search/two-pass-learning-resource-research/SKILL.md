# 两轮学习资料搜索与推荐 Skill

## 适用场景

当用户提出以下类型的请求时使用本 Skill：

- 搜索某门技术、课程或工具的优质教材；
- 优先寻找中文文字稿、在线文档、PDF、教程专栏或官方手册；
- 不希望只依据搜索排名或宣传页推荐；
- 希望结合知乎、V2EX、CSDN 评论区、技术论坛、Reddit、Hacker News 等使用者经验进行二次筛选；
- 希望判断“哪一本适合作为主教材、哪一些只适合查阅或排错”。

典型主题：STM32 HAL、ROS2、Linux、C/C++、Python、控制理论、模电、算法、EDA、嵌入式工具链等。

---

## 核心原则

必须把搜索分成相互独立的两轮：

1. **第一轮：全力搜索教材本身。** 只评估资料的完整性、准确性、结构、可复现性和适配程度，不因社区口碑先入为主。
2. **第二轮：搜索真实学习经验。** 在第一轮候选池基础上，查找学习者的实际使用反馈、踩坑、适用人群、版本问题和完成后的效果，再重新排序。

不得把“第一轮搜索到的第一个知名教程”直接当作最终推荐。

---

## 输入信息

从用户请求和上下文中提取：

- 学习主题；
- 目标语言，例如中文；
- 资料形态，例如文字优先、在线文档、PDF；
- 学习者当前水平；
- 使用的平台、芯片、版本或工具链；
- 学习目的，例如入门、比赛、项目开发、工程进阶；
- 是否接受英文官方资料作为核对来源。

信息不足时不要停下搜索。根据现有信息做合理假设，并在结果中明确假设。

# 第一轮：全力搜索教材

## 1. 建立搜索词矩阵

围绕主题组合以下关键词，不要只使用一个查询：

### 教材形态

- 中文教程
- 中文文档
- 在线教材
- 开发指南
- 实战指南
- 入门手册
- PDF
- GitBook
- Read the Docs
- GitHub README
- 系列教程
- 学习笔记
- 官方示例
- 用户手册
- API 手册

### 质量与结构

- 完整目录
- 配套代码
- 示例工程
- 从零开始
- 系统教程
- 章节
- 原理
- 源码分析
- 工程化
- 最小例程

### 版本与目标平台

加入芯片、框架、操作系统、库版本、开发板、IDE 等限定词。

示例：

```text
STM32F103 HAL 中文 在线教材 配套代码
STM32CubeF1 HAL 开发指南 PDF 中文
STM32 HAL 系列教程 GPIO UART DMA ADC 定时器
site:github.com STM32F1 HAL 中文 教程
site:doc.embedfire.com STM32F103 HAL
```

## 2. 优先搜索以下来源层级

第一轮应尽可能覆盖不同来源，而不是只搜索博客：

1. 官方文档、官方软件包、官方示例；
2. 出版物或硬件厂商的系统教材；
3. 可连续阅读的在线文档；
4. GitHub 上结构完整的教程仓库；
5. 高质量个人系列教程；
6. CSDN、博客园、知乎专栏等零散文章。

## 3. 为每个候选建立资料卡

至少记录：

- 名称；
- 作者或机构；
- 链接；
- 资料形式；
- 目标版本和硬件；
- 目录覆盖范围；
- 是否有配套代码；
- 是否讲原理；
- 是否只讲配置步骤；
- 是否便于全文搜索；
- 更新时间；
- 明显缺点；
- 适合的学习阶段。

## 4. 第一轮评分维度

每项按 1-5 分评价：

- **系统性**：能否形成完整学习主线；
- **准确性**：是否有官方依据、源码或参考手册支撑；
- **可复现性**：是否有完整代码、配置和实验现象；
- **原理深度**：是否解释为什么，而不只是点击和调用；
- **检索体验**：能否全文搜索、复制、跳转章节；
- **目标适配**：是否匹配用户的芯片、版本和水平；
- **时效性**：版本是否仍然适用；
- **AI 友好度**：是否方便截取、引用、比较和整理为个人教材。

第一轮结束时只形成“候选池”和初步评价，不做最终结论。

## 5. 第一轮最低覆盖目标

条件允许时，应搜索到：

- 2 个以上官方或准官方资料；
- 2 个以上系统教材；
- 2 个以上个人教程或学习笔记；
- 1 个以上函数速查、API 或示例库；
- 1 个以上工程化或进阶资料。

如果资料稀缺，应明确说明实际找到的数量，不得虚构候选。

---

# 第二轮：结合学习者经验复筛

## 1. 只围绕第一轮候选搜索反馈

对每个主要候选分别搜索：

```text
候选名称 学习体验
候选名称 适合新手吗
候选名称 学完
候选名称 踩坑
候选名称 缺点
候选名称 过时
候选名称 配套代码 问题
候选名称 知乎
候选名称 V2EX
候选名称 论坛
候选名称 评论
```

同时搜索主题层面的学习方法讨论：

```text
主题 文档还是视频 知乎
主题 文字教程 学习经验
主题 入门 路线 论坛
主题 官方文档 看不懂
主题 tutorial hell
主题 只会照抄 教程
```

## 2. 重点来源

中文优先搜索：

- 知乎问题与回答；
- V2EX；
- ST 中文社区、厂商论坛、电子工程论坛；
- CSDN 评论区与系列文章评论；
- 博客园评论；
- GitHub Issues、Discussions；
- 开源社区、开发板论坛；
- B站视频评论区（能可靠访问时）。

必要时补充：

- Reddit；
- Hacker News；
- Stack Overflow；
- 官方社区论坛。

## 3. 识别反馈质量

将反馈分成三档：

### A 级：第一手、具体、可验证

具备以下特征：

- 明确说明自己使用过；
- 描述学习阶段或项目；
- 给出具体章节、代码、版本或问题；
- 同时提到优点和缺点；
- 没有明显推广链接。

### B 级：一般经验判断

例如“新手适合视频，中级以后适合文档”，有参考价值，但缺乏候选教材的具体使用细节。

### C 级：低可信或推广性质

包括：

- 带邀请码、课程销售或返利；
- 只有“非常好”“效率提高十倍”之类笼统评价；
- 明显复制宣传文案；
- 未表明是否实际使用；
- 用单一失败案例否定整套教材。

C 级反馈可以提及，但不能成为主要推荐依据。

## 4. 主动搜索负面证据

不得只搜索“推荐”和“好用”。必须专门查找：

- 版本过旧；
- 例程无法运行；
- 只会照抄；
- 缺少原理；
- 与自家开发板绑定；
- 工程结构不适合长期项目；
- 章节跳跃；
- 配套代码缺失；
- 新版 IDE 或库不兼容；
- 学完仍无法独立开发。

如果没有找到明确负面经验，应写“未检索到足够负面反馈”，不能写成“没有缺点”。

## 5. 区分教材评价与学习方式评价

例如以下反馈不能混为一谈：

- “这本教材目录完整”是对资料本身的评价；
- “我不喜欢看视频”是对媒介偏好的评价；
- “跟着教程能运行，但不会独立写”是对学习方法的评价；
- “旧版 CubeMX 界面不同”是版本兼容问题。

输出时要分别说明。

## 6. 第二轮重排评分

建议按以下权重重新评分：

- 教材系统性：20%
- 权威性与准确性：20%
- 代码和实验可复现性：15%
- 与用户目标适配：15%
- 学习者真实反馈：15%
- 文字检索与 AI 处理友好度：10%
- 版本时效性：5%

权重可以随用户目标调整：

- 入门者提高“系统性”和“适配”；
- 项目开发提高“可复现性”和“时效性”；
- 工程进阶提高“源码、架构和官方资料”；
- 考试学习提高“结构、例题和练习”。

# 最终推荐结构

最终回答必须体现“两轮搜索”，不能只给一个链接列表。

## 第一部分：第一轮候选池

简要说明搜到了哪些类别和代表资料：

- 系统主教材；
- 官方资料；
- 在线教程；
- 个人笔记；
- API 速查；
- 工程进阶资料。

对每个候选说明其本身的优势、局限和适配对象。

## 第二部分：学习者经验

按主题总结真实反馈：

- 学完以后能做到什么；
- 新手最常卡在哪里；
- 哪些内容容易照抄；
- 版本和配套代码是否有问题；
- 文字、视频和官方文档各自的适用阶段；
- 哪些评价带有推广性质。

尽量转述，不大段复制评论。需要直接引用时保持简短，并标注来源。

## 第三部分：二次筛选后的最终排名

不要只排第一、第二、第三。应给每份资料一个明确角色：

- **主教材**：负责建立完整知识体系；
- **权威核对**：官方文档、源码和示例；
- **工程进阶**：学习架构、调试和项目组织；
- **问题检索**：博客、论坛和踩坑文章；
- **函数速查**：API、宏、参数和最小例程。

## 第四部分：推荐的使用方法

给出实际学习闭环，例如：

```text
阅读主教材对应章节
→ 自己建立最小工程
→ 不复制完整答案，填写关键代码
→ 修改参数观察结果
→ 打开官方源码核对
→ 遇到具体问题再搜索博客和论坛
→ 用自己的话写一页总结
```

---

# 事实与引用要求

- 资料目录、版本、更新时间、示例数量等事实必须来自实际页面；
- 学习者体验必须来自真实讨论，不得把自己的推断写成“网友普遍认为”；
- 不得捏造知乎回答、评论、点赞数或用户原话；
- 只找到宣传文章时，应明确标记“带推广性质”；
- 同一结论尽量由不同平台交叉支持；
- 论坛回复是个人经验，不应表述为严格研究结论；
- 评价资料是否过时时，要同时检查页面更新时间和其针对的技术版本；
- 当官方资料与博客冲突时，以当前官方文档、源码和可运行实验为准。

---

# 常见错误

禁止以下做法：

1. 搜到知名品牌教材后立即停止；
2. 把搜索排名当作质量排名；
3. 只看教材首页，不查看目录和样章；
4. 只查正面推荐，不查缺点与版本问题；
5. 把厂商宣传页当作学习者反馈；
6. 罗列十几条链接，却不给资料角色和学习顺序；
7. 用一条匿名评论推翻一整套教材；
8. 推荐多个高度重复的主教材，让用户同时学习；
9. 把 CSDN 等博客平台整体判定为“好”或“差”；
10. 让 AI 生成的无来源内容高于官方资料和实测结果。

# 推荐判断准则

最终通常只选择一套主线，避免资料切换成本。

理想组合是：

```text
一套系统中文主教材
+ 当前官方文档、源码和示例
+ 一套工程进阶资料
+ 若干按问题检索的踩坑文章
```

推荐语言示例：

> 第一轮从完整度看，A、B、C 都进入候选；第二轮结合学习者反馈后，A 更适合作为主教材，B 更适合对应自家开发板，C 适合已经掌握基础后的工程进阶。官方资料不承担中文入门教学，但作为最终核对来源不可替代。

# HAL 主题的示例执行

用户请求：

```text
搜索 STM32F103 HAL 库的优质中文文字教材，先全面找教材，再结合知乎和类似网站的学习经验二次筛选。
```

第一轮应搜索：

- STM32F103 HAL 中文在线教材；
- 野火、正点原子、安富莱相关资料；
- STM32CubeF1 官方包和示例；
- GitHub 中文教程仓库；
- CSDN、博客园连续系列；
- HAL API 中文速查资料。

第二轮应搜索：

- 野火 HAL 学习体验、缺点、版本；
- 正点原子 HAL 学习体验、代码风格；
- 安富莱教程适合新手吗；
- HAL 文档还是视频；
- STM32 教程照抄、独立开发、面向百度编程；
- CubeMX 版本变化和旧教程兼容问题；
- 官方 HAL 源码和 Examples 的实际使用方式。

最终输出角色通常应区分为：

- 主教材；
- 官方权威字典；
- 工程进阶；
- 问题检索；
- 函数速查。
